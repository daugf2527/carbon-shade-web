## 二、修订后的 Stage 1.5 P0 任务（按 Stage 2 阻塞度排序）

> **共识门槛**：Stage 2（13-system engine layer）启动前必须全部完成，否则会造成"runtime 拿不到 mob HP"、"技能没有 cooldown"、"觉醒技能挂在角色身上但无 ID 引用"等基本残废。

### P0-1. MobParser 全字段扩展（**最高优先**）

**问题**: 当前 MobParser 只读 7 个字段，且 hpMax 永远 null。怪物 HP 算不出 → 13-system 战斗循环跑不起来。

**改动**:
- 加 `abilityCategory: Record<string, number>` — 解析形如 `[hp max] * 80, [equipment_physical_attack] * 80` 的百分比缩放表
- 加 `level: [min, max]` — 怪物等级范围
- 加 `attackDelay: number` — 两次攻击最小间隔 ms
- 加 `moveSpeed: [x, y]` — 双维移动速度
- 加 `hitRecovery: [base, scale]` — 硬直恢复 ms
- 加 `width: [w, h]` — 碰撞体宽高
- 加 `attackKind: PvfAttribute[]` — 12-84 个不等 float/int（结构保留，下游 unpack）
- 加 `stuckbonus: number[]` — 4 元素霸体加成
- 加 `dropItems: Array<{id, rate}>` — item 表 unpack
- fix `weight` 用 vectorFact 取双维（当前 firstNumberFact 丢第二维）
- 同步加 validator schema 字段

**验收**:
- baseline goblin.json `abilityCategory.hp_max === 80`
- baseline goblin.json `effectiveHpMax === abilityCategory.hp_max * baseHpReference`（baseHpReference 由 ChrParser growth.hpMax 推算或 manifest 注入）
- regression test 覆盖 5+ 怪物（cowardgoblin/goblinhodor/boss 等）

**预计**: 1 个 opus agent 1 天 + 重跑 baseline

---

### P0-2. SklParser 12 个 raw section typed 化

**问题**: 当前 SklParser 只 typed 化了 cancel 系列 + 基本元数据（skillType / purchaseCost / requiredLevel 等）。12 个最常用 section 仍在 raw 里没结构化。Stage 2 技能系统跑不起来（拿不到 CD/MP/施放时间/按键序列）。

**12 个 section（按 raw 出现频率 + Stage 2 紧急度排序）**:
| section | swordman 命中 | 含义 | Stage 2 系统 |
|---------|--------------|------|-------------|
| `[icon]` | 205/205 | 图标 `[atlas, frame, lit_atlas, lit_frame]` | UI |
| `[level property]` | 155/205 | 逐级数值公式 (9 values) | Skill 升级引擎 |
| `[pre required skill]` | 103/205 | 前置技能 `[id, level]` | SkillTree |
| `[command]` | 88/205 | 按键序列 token | InputLayer |
| `[feature skill index]` | 50/205 | 技能引擎分类 ID | SkillTree |
| `[consume item]` | 37/205 | 消耗道具 | ResourceSystem |
| `[skill command advantage]` | 35/205 | 命令优先级窗口 | InputLayer |
| `[consume mp]` | 32/205 | MP 消耗 | ResourceSystem |
| `[level info]` | 27/205 | 逐级具体数值表 | Skill 升级引擎 |
| `[cool time]` | 22/205 | 冷却时间 `[dungeon_ms, pvp_ms]` | ResourceSystem |
| `[casting time]` | 20/205 | 施放时间 `[base_ms, lvl20_ms]` | AnimPlayback |
| `[maintain mp]` | 4/205 | 持续 MP 消耗（toggle） | ResourceSystem |

**改动**:
- 给 SklDef 加 12 个 typed 字段（null safe — 不是所有技能都有）
- SklParser 解析逻辑 + tier-3 标注（不确定的字段）
- validator schema 同步加字段
- 同步给"未来扩展"留 raw escape hatch

**验收**:
- baseline swordman.json typed-field 覆盖率 20% → 80%（按 raw section 出现的字段中已 typed 的比例计算）
- 关键技能（hardattack/dashattack/asurakazan）的 cool time + consume mp + casting time 都非 null

**预计**: 1 个 opus agent 1 天

### P0-3. ChrParser 4 个 awakening sections

**问题**: raw .chr 实存 4 个 awakening section（awakening 1/2/name/skill），parser 完全没读。觉醒系统挂在角色身上但 ID/名字都没出口。

**改动**:
- ChrDef 加 `awakening: { skillSlots, names, tier1SlotCounts, tier2SlotCounts }`
- ChrParser 加 parseAwakening 函数
- validator ChrDefSchema 配套加 awakening schema
- 修 h15-export-probes test fixture 加 awakening 默认值

**注意**:
- raw awakening name 在 70-cap 数据里全是 link("") 空字符串（觉醒名在 stringtable.bin 里）
- awakening 1/2 全空，可能是高级觉醒槽位 placeholder
- 不要臆测 sub-job 维度语义 — 保守保留原始数据 + Tier-3 mark

**验收**:
- baseline swordman.json `awakening.skillSlots.length === 10`，非空 entries 至少 4 个，例如 `[[91,1], [], [89,1], ...]`

**预计**: 主对话 30 min（已尝试 1 次因 validator 冲突回退）

---

### P0-4. AniDef inline 进 EXPORT（API 已就绪）

**问题**: RuntimeExporter 已有 `aniDefs?: ReadonlyArray<AniDef>` 可选参数 + `playerAnims: Record<string, AniDef>` 字段，但默认不传 → animations 字段总是 `{}`。Stage 2 战斗运行时拿不到逐帧 hitbox。

**改动**:
- baseline 脚本（`scripts/stage1-baseline.mjs`）跑完 chr/skl/atk/mob 后，遍历 chr.motionRefs / mob.animationRefs，把对应 .ani 也提取 + 解析
- 把 parsed AniDef[] 喂给 RuntimeExporter.exportRuntimeShards 的 aniDefs 参数
- 至少覆盖 11 个 player chr 的核心 motion（attack/jump/dash/damage/down/getup 等 ~24 个）

**验收**:
- baseline swordman.json `animations` 字段非空，含 attack1/attack2/attack3/jumpattack/dashattack 等
- regression test 验证 swordman.animations["attack3"].frames[2].attackBoxes 非空

**预计**: 主对话 1-2 小时

## 五、与 Stage 2 brainstorm 的对接

原 [Stage 2 brainstorm](2026-05-24-stage2-brainstorm.md) 提出 13 个 system layer。验证后梳理"哪些 system 当前数据已就绪 / 哪些 system 必须等 Stage 1.5 P0 完成"：

| Stage 2 system | 当前数据状态 | 需要的 Stage 1.5 任务 |
|---------------|------------|--------------------|
| 1. Combat math (damage formula) | ✅ ChrDef physical/magical attack + AtkDef damageBonus 已 typed | — |
| 2. Skill resource (CD/MP) | ❌ 12 raw section 未 typed | P0-2 SklParser |
| 3. Input command | ❌ `[command]` raw 未 typed | P0-2 |
| 4. AI loop | ⚠️ MobDef.category 已有，但行为常数缺 | P0-1 MobParser |
| 5. Animation playback | ❌ AniDef 未 inline | P0-4 |
| 6. Hit detection | ⚠️ atk[]/dmg[] 6-int 已 typed，但**普通攻击 attack1.ani 无 atk hitbox**（须靠 .atk 反推或 dmg 框双向判定） | 设计抉择（见下） |
| 7. Reaction (hitstun) | ⚠️ Tier-3 baseline，DNF.exe 硬编码无法提取 | local_baseline + manual 调参 |
| 8. Resource pool | ❌ MP/HP 上限缺（mob hpMax / chr awakening） | P0-1 + P0-3 |
| 9. Status effect | ✅ status manifest 已有（master 分支历史成果） | — |
| 10. Replay / determinism | ✅ kernel 层已就绪 | — |
| 11. Camera | ✅ MapDef 3-level scroll 已 typed | — |
| 12. UI (icon/HP bar) | ⚠️ skill `[icon]` 未 typed | P0-2 |
| 13. Sound | ⚠️ AtkDef.hitWav + ChrDef.weaponWav 已 typed，但音频文件路径未做 NPK resolve | 不阻塞 |

### 关键设计抉择（Stage 1.5 之后立即面对）

**普通攻击 attack1.ani 无 atk hitbox 的处理**：
- **选项 A**: 引擎运行时根据 .atk 文件的 attackInfo 配合 dmg 框反推判定范围
- **选项 B**: 所有"无 atk hitbox 的 .ani 帧"都用 dmg 框做双向判定（角色受击 + 角色出招的同一 box）
- **选项 C**: 在 ChrParser 阶段，对每个 motion 的 .ani 自动检查"有 atk 帧"是否存在；如果都没有，从对应 attackInfo[0] .atk 反推默认 hitbox

**决定（2026-05-26）：选 A — `.atk` 反推**。Stage 2 hit detection 运行时按 chr.motionRefs[motionName] 找到对应 .atk，从 AtkDef 的物理参数（liftUp/pushAside/knuckBack）+ 角色 widthBox 推导判定几何。后续如果 AtkDef 需要加显式 hitbox 字段，再做 schema 扩展。AniDef 的 atk[] 仍然作为"权威优先级"，存在则覆盖 .atk 反推。

不论选哪种，**这是 Stage 1.5 后 Stage 2 的第一个抉择**，不再属于数据层补件。
