## 5. 不能走偏的护栏

### 5.1 五条铁律（违反就停下重评估）

1. **每个 Day 完成才能进下一个 Day**。验证标准：本文档 §4 每个 Day 末尾的"verification"清单全部通过 + `completion.mjs` 对应 Day 转绿。
2. **不许跳 VALIDATE 直接进 LOAD**。design §2.2 明确 PARSE → **VALIDATE** → LOAD。validator.ts 必须存在且接进 pipelineRunner。
3. **不许跳 smoke test 直接 commit Day 13**。`tests/smoke/full-pipeline.test.ts` 必须跑通真 PVF。
4. **任何"我们先简化下"的提议**——拒绝，引 [[feedback-dnf-pve-scope-only]]："不接受工程经济性论据压缩 13 system / Stage 1 设计"。
5. **任何"先去做 Stage 2 引擎层"的提议**——拒绝。Stage 1 没完成（Day 11-17）之前不进 Stage 2 系统层实施。允许 **brainstorm** Stage 2 design 并行（design 只产文档）。

### 5.2 工作流强制使用 `/closed-loop`

每个 Day 完成时，**必须**跑一次 `/closed-loop`：
1. dispatch audit agents on the topic（参考 `.claude/skills/audit/topics.json`）
2. audit-verify
3. 主 Claude 核 verdict
4. checkpoint B 让用户决定修哪些
5. 修复 + 5/6/7/8 验证 + 9 commit

**不许**手工跑 audit / 手工核 / 手工 commit。流程已经固化成 skill，每次 Day 完成跑一次保证不漏。

### 5.3 临时增项的判断标准

不在 Day plan 但发现的工作（如今天的 C++ memory-safety audit）：

| 类型 | 处理 |
|---|---|
| 当前 Day 的 blocker（不修就过不去）| 立刻修，作为当前 Day 的 sub-item |
| 别的 Day 的内容（如 Day 12 看到了 EXPORT 的问题）| 写进对应 Day 的 prep notes，**不立刻做** |
| Stage 2 相关 | 写进 backlog（[[combat-lab-dnf-alignment-pivot-2026-05-21]] 已划为 Stage 1 后） |
| 工作流基础设施（audit-verify / completion 这种）| 一旦发现"我们这次审用得上"，立刻建（用户 2026-05-23 已示范） |
| 性能优化 | 除非破坏可用性（如今天 `--list` 10s 还算能用），否则进 backlog |

### 5.4 Memory 与 docs 同步责任

每完成一个 Day:
- 更新 `MEMORY.md`（如有教训）
- 更新本文档（§4 对应 Day 加 ✅ + 落档 commit hash）
- 更新 `CLAUDE.md` 现状段
- `npm run completion` 应该自动反映 Day 进度
